2
Same moment → MOBILE_DEVICE row inserted · SESSION_LEG leg #1 created (participant_type=customer, customer_id set, prev_leg_id=NULL)
3

7

Session-Centric Design — v3

CHAT_SESSION remains the root. The key additions are SESSION_LEG (tracks every participant segment within a session) and TRANSFER_EVENT (full audit of every handoff). All IDs changed from UUID to BIGINT for simpler queries and better join performance.

🏛️ Full Table Hierarchy
CHAT_SESSION ← ROOT · session_id BIGINT · no outward FK
│
├──MOBILE_DEVICE· captured at start · 1:1
├──UCID_MAPPING· async, inserted later · 1:1
├──DISCONNECT_EVENT· written at session end · 1:1
├──LICENSE_EVENT· session_id nullable, independent audit · 1:N
└──SESSION_LEG· one row per participant segment · 1:N
│ ↺ prev_leg_id (self-ref) → transfer chain
├──TRANSFER_EVENT· from_leg + to_leg audit · 1:N
├──CHAT_MESSAGE· messages now link to LEG · 1:N
└──LEG_DISCONNECT· per-leg disconnect details · 1:1 per leg

NEW TABLE

💬 CHAT_SESSION ROOT · no FK outward ▼

The root table — generated first, referenced by every other table. Contains the top-level session state. No foreign key outward so it can always be inserted first without depending on any other record.

PK session_idBIGINT
Auto-generated identity. The anchor ID — every other table references this. Never set by application code.
start_timeTIMESTAMP NOT NULL
Set to NOW() when the row is created. Records when the session began, not when the first message arrived.
end_time nullableTIMESTAMP
NULL while the session is active. Set when the session terminates for any reason — clean close, drop, or license kill.
statusENUM NOT NULL
Quick state check without joining child tables. Values: active (ongoing) · ended (clean close) · dropped (connection lost) · license_exceeded (killed by license cap).
channelVARCHAR(32)
The medium the chat arrived on — e.g. mobile_app, web, api. Useful for channel-level analytics and routing rules.
created_atTIMESTAMP NOT NULL
Server-side row creation time. Separate from start_time to handle processing delay between event and DB write.
metadataJSONB
Flexible bag for any attributes that don't warrant a dedicated column — region, queue_id, agent_group, priority, campaign tags.
📌 Recommended Indexes
IDXidx_cs_status
status
Fast filter by session state. Most operational queries filter on status=active or status=license_exceeded.
IDXidx_cs_created
created_at DESC
Time-range reporting — e.g. "all sessions in the last 24 hours". DESC because most queries look at recent data first.
IDXidx_cs_status_ts
(status, created_at DESC)
Compound index for "all active sessions today" — covers both filter and sort in a single index scan, no table lookup needed.
Why no FK outward? If CHAT_SESSION pointed to other tables, those records would need to exist before the session could be created. Being FK-free means the session is always created first and all children attach after — synchronously or asynchronously.
📱 MOBILE_DEVICE IDENTITY · 1:1 · written at session start ▼

Written at the same time as CHAT_SESSION. Captures device identity — mobile_id, platform, OS version, app version, and a hardware fingerprint that persists across app reinstalls.

PK device_idBIGINT
Auto-generated identity. Internal key — not the app's mobile_id.
FK session_idBIGINT NOT NULL
Links to the root CHAT_SESSION. ON DELETE CASCADE — device row is removed if session is deleted.
UK mobile_idVARCHAR(64) NOT NULL UNIQUE
The app-assigned device identifier. UNIQUE ensures one device record per mobile_id. Use to find all sessions from a single device.
platformVARCHAR(32)
iOS or Android. Enables per-platform analytics.
os_versionVARCHAR(32)
OS version at session time. Useful for debugging OS-specific bugs.
app_versionVARCHAR(32)
App build version at session time. Critical for correlating bugs with specific releases.
device_fingerprintVARCHAR(128)
Hardware-level hash. Persists across reinstalls. Links same physical device even if mobile_id changes.
captured_atTIMESTAMP NOT NULL
When this device row was written.
📌 Recommended Indexes
IDXidx_md_session
session_id
FK index — required for JOIN from chat_session to mobile_device. Without this every JOIN scans the full table.
IDXidx_md_mobile_id
mobile_id
Supports "find all sessions from this device". Also backs the UNIQUE constraint.
Why not merged into CHAT_SESSION? Device info is a different concern from session state. Keeping it separate avoids wide rows on the hot CHAT_SESSION table, and device data could be reused across future designs.
🔗 UCID_MAPPING ASYNC · 1:1 · inserted later ▼

Inserted asynchronously after the session is created, once the CRM or telephony system resolves the UCID. This row may not exist at session start — session creation is never blocked waiting for UCID resolution.

PK ucid_map_idBIGINT
Auto-generated identity.
FK session_idBIGINT NOT NULL
Links to CHAT_SESSION. Inserted later when CRM resolves the UCID.
UK ucid_valueVARCHAR(128) NOT NULL UNIQUE
The UCID from telephony/CRM. UNIQUE — each UCID maps to exactly one session.
mobile_idVARCHAR(64)
Denormalized copy of mobile_id for direct UCID→device lookup without an extra JOIN.
resolved_atTIMESTAMP NOT NULL
When the UCID was resolved. Gap between session.start_time and resolved_at = UCID resolution latency.
source_systemVARCHAR(64)
Which system provided the UCID — e.g. "genesys", "avaya", "salesforce".
resolution_statusENUM NOT NULL
resolved / failed / pending. Failed = UCID lookup attempted but CRM couldn't match it.
notesTEXT
Free-text for resolution errors, retry info, or context from the resolving system.
📌 Recommended Indexes
IDXidx_um_session
session_id
FK index and used in "find unresolved sessions" query (LEFT JOIN WHERE ucid_map_id IS NULL).
IDXidx_um_value
ucid_value
Backs the UNIQUE constraint. Enables fast UCID→session lookups from telephony side.
IDXidx_um_mobile_id
mobile_id
Enables UCID→device lookups without going through the session.
Why async? UCID from telephony systems is often not available when a chat session starts. Separate table + async insert means session creation is never blocked. LEFT JOIN handles the "not yet resolved" state cleanly in queries.
🦵 SESSION_LEG ⭐ NEW · 1:N · core routing ▼

The most important new table. One row per participant segment — every customer, agent, supervisor, or bot active in a portion of the session gets its own leg row. prev_leg_id is a self-referencing FK that chains all legs into a transfer path. Messages and transfer events both reference this table.

PK leg_idBIGINT
Auto-generated identity. Referenced by TRANSFER_EVENT, CHAT_MESSAGE, and LEG_DISCONNECT.
FK session_idBIGINT NOT NULL
Links this leg to its parent session. All legs in a session share this session_id.
leg_sequenceINT NOT NULL
1 = first leg, 2 = second, etc. UNIQUE per session. Provides deterministic ordering without parsing timestamps.
participant_typeENUM NOT NULL
customer / agent / supervisor / bot. Identifies which ID column is populated and the participant's role.
customer_id nullableBIGINT
Set when participant_type=customer. NULL otherwise.
agent_id nullableBIGINT
Set when participant_type=agent. NULL otherwise.
supervisor_id nullableBIGINT
Set when participant_type=supervisor. NULL otherwise.
bot_id nullableBIGINT
Set when participant_type=bot. NULL otherwise.
FK↺ prev_leg_id nullable · self-refBIGINT
Self-referencing FK. NULL for the first leg. Points to the leg active before this one — walk backwards to reconstruct the full transfer path.
transfer_reasonVARCHAR(128)
Quick note on why this leg was created — "customer escalation", "bot handoff", "queue overflow".
leg_startTIMESTAMP NOT NULL
When this participant became active.
leg_end nullableTIMESTAMP
NULL while active. Set when participant leaves — transferred, completed, or dropped.
leg_statusENUM NOT NULL
active / completed / transferred / dropped.
📌 Recommended Indexes
IDXidx_sl_session
session_id
Primary FK index. Critical — without this, fetching all legs for a session scans the full table.
IDXidx_sl_customer (filtered)
customer_id WHERE customer_id IS NOT NULL
Find all sessions for a customer. Filtered — only indexes rows where the column is populated.
IDXidx_sl_agent (filtered)
agent_id WHERE agent_id IS NOT NULL
Find all sessions an agent handled. Essential for agent dashboards and workload queries.
IDXidx_sl_supervisor (filtered)
supervisor_id WHERE supervisor_id IS NOT NULL
Supervisor workload and escalation history.
IDXidx_sl_bot (filtered)
bot_id WHERE bot_id IS NOT NULL
Bot performance analytics — sessions handled, handoff rate.
IDXidx_sl_prev_leg (filtered)
prev_leg_id WHERE prev_leg_id IS NOT NULL
Forward traversal of transfer chain — "what leg came after leg X?". Filtered because Leg 1 always has prev_leg_id=NULL.
Why separate ID columns per role? Each role has its own namespace — customer_id=5 and agent_id=5 are different people. Separate columns make each role's ID self-contained, independently indexable, and unambiguous in JOINs to your identity tables (customers, agents, bots, supervisors).
🔄 TRANSFER_EVENT ⭐ NEW · 1:N · transfer audit ▼

Full audit of every transfer in a session. While prev_leg_id shows the chain structure, TRANSFER_EVENT captures the full details: who initiated it, what type it was, when it completed, and whether it succeeded or failed.

PK transfer_idBIGINT
Auto-generated identity.
FK session_idBIGINT NOT NULL
Denormalized for direct session-level transfer queries without joining through legs.
FK from_leg_idBIGINT NOT NULL
The leg that initiated the transfer — the departing participant's SESSION_LEG row.
FK to_leg_id nullableBIGINT
NULL when first recorded (status=pending). Updated once the receiving leg exists. NULL = transfer in progress.
transfer_typeENUM NOT NULL
agent_to_agent / agent_to_supervisor / bot_to_agent / bot_to_supervisor / customer_to_agent / agent_to_bot.
transfer_reasonVARCHAR(128)
Why the transfer happened — "customer requested human", "bot confidence low", "specialist needed".
initiated_by nullableENUM
customer / agent / supervisor / bot / system. Was this the customer asking for a human, or the bot deciding to escalate?
transferred_atTIMESTAMP NOT NULL
When the transfer was initiated.
completed_at nullableTIMESTAMP
When the receiving participant accepted. NULL while pending or if failed.
statusENUM NOT NULL
pending / completed / failed.
📌 Recommended Indexes
IDXidx_te_session
session_id
Get all transfers for a session — used in session reconstruction and timeline views.
IDXidx_te_from_leg
from_leg_id
Forward traversal: "what transfer came from leg X?"
IDXidx_te_to_leg (filtered)
to_leg_id WHERE to_leg_id IS NOT NULL
Reverse lookup: "what transfer created leg X?". Filtered because to_leg_id is NULL for pending/failed transfers.
Why both prev_leg_id and TRANSFER_EVENT? They are complementary. prev_leg_id is the structural link — cheapest for JOINs and chain traversal. TRANSFER_EVENT is the audit record — timing, failure state, who initiated, gap between "transfer started" and "transfer completed". You need both.
✉️ CHAT_MESSAGE UPDATED · 1:N · now links to leg ▼

Raw message store — one row per message, high volume. In v3 messages carry both session_id and leg_id, linking each message to the exact participant segment that sent it. sequence_no ensures correct ordering regardless of timestamp accuracy.

PK message_idBIGINT
Auto-generated identity.
FK session_idBIGINT NOT NULL
Direct FK to the session — kept so you can fetch all messages in a session without joining through legs.
FK leg_idBIGINT NOT NULL
NEW in v3. Links to the active SESSION_LEG — tells you exactly which agent/bot/supervisor sent this message.
sequence_noINT NOT NULL
Monotonically increasing per session. UNIQUE per (session_id, sequence_no). Guarantees correct conversation order even when timestamps are identical or arrive out of order.
sender_typeENUM NOT NULL
customer / agent / supervisor / bot. Quick filter without joining to session_leg.
sender_id nullableBIGINT
The specific external ID of the sender. Together with sender_type, fully identifies who sent each message.
message_typeVARCHAR(32)
text / file / image / event / system. Determines how to interpret content and raw_payload.
content nullableTEXT
Extracted text. NULL for file/image types — use raw_payload for those.
raw_payloadJSONB
Complete original message object. Never transformed — preserves all metadata. Enables reprocessing if parsing logic changes.
sent_at nullableTIMESTAMP
The sender's claimed timestamp. May be unreliable (client clock skew). Use sequence_no for ordering.
received_atTIMESTAMP NOT NULL
Server-side receipt timestamp — authoritative and always present. Used for latency calculations.
📌 Recommended Indexes
IDXidx_cm_session_seq
(session_id, sequence_no ASC)
The most critical index. Used in every "get conversation history" query. Compound covering index — fetches messages in order in a single scan.
IDXidx_cm_leg
leg_id
Get all messages from a specific leg — used for per-participant message views and agent performance metrics.
IDXidx_cm_sender
(sender_id, sent_at DESC)
Find all messages from a specific agent/bot across all sessions. DESC for most-recent-first.
Why keep session_id on messages when we have leg_id? Performance. Fetching all messages by leg_id would require joining through session_leg first. With session_id directly on the message, the full conversation history is one index scan on idx_cm_session_seq — the most common query pattern.
⚡ DISCONNECT_EVENT REASON · 1:1 · written at session end ▼

Written exactly once when the session terminates for any reason. The is_license_limit and tcomm_triggered boolean flags are the core mechanism for identifying silent license-kill events — the original problem this schema was designed to solve.

PK disconnect_idBIGINT
Auto-generated identity.
FK session_idBIGINT NOT NULL UNIQUE
Links to the session. UNIQUE enforces exactly one disconnect record per session.
reason_codeVARCHAR(64)
Machine-readable code — e.g. LICENSE_EXCEEDED, USER_CLOSED, NETWORK_DROP, TIMEOUT, AGENT_CLOSED.
reason_detail nullableTEXT
Human-readable description. May include error messages or agent notes.
is_license_limitBOOLEAN NOT NULL
KEY FIELD. TRUE when the disconnect was caused by the license concurrent session cap. The primary flag answering: "was this session killed by the license limit?"
tcomm_triggeredBOOLEAN NOT NULL
KEY FIELD. TRUE when tcomm itself initiated the disconnect. is_license_limit=TRUE + tcomm_triggered=TRUE = the "silent license kill" scenario.
error_code nullableVARCHAR(32)
Low-level transport error code — e.g. HTTP 503, WebSocket 1001. Separate from the higher-level reason_code.
disconnected_atTIMESTAMP NOT NULL
When the disconnect event was recorded. Session duration = disconnected_at − session.start_time.
raw_payload nullableJSONB
Complete original disconnect signal — never transformed. Preserves all debug info.
📌 Recommended Indexes
IDXidx_de_license_limit
(is_license_limit, disconnected_at DESC)
Primary analytics index. Enables "find all license-killed sessions" with time range. DESC for recent-first.
IDXidx_de_tcomm
(tcomm_triggered, disconnected_at DESC)
Filter sessions where tcomm drove the disconnect. Combined with is_license_limit isolates the exact license-kill pattern.
IDX(auto) session_id
via UNIQUE constraint
The UNIQUE(session_id) constraint automatically creates a unique index — no separate FK index needed.
Why boolean flags instead of just reason_code? reason_code is flexible text — string matching is slow, error-prone, and breaks if wording changes. Boolean flags are indexed, unambiguous, and database-native. WHERE is_license_limit = TRUE AND tcomm_triggered = TRUE is faster and more reliable than WHERE reason_code LIKE '%LICENSE%'.
🔑 LICENSE_EVENT AUDIT · independent · session_id nullable ▼

Independent audit log for all license-related events. The nullable session_id is the critical design — it allows recording license events that fire before a session is ever created. The partial index on WHERE session_id IS NULL makes pre-session blocks fast to query.

PK license_event_idBIGINT
Auto-generated identity.
FK session_id nullableBIGINT
Nullable FK to CHAT_SESSION. NULL when the license event fires before a session could be created. ON DELETE SET NULL preserves the event even if its session is deleted.
mobile_id nullableVARCHAR(64)
Device ID of the connection attempt. Captured directly for pre-session events where no session_id (and therefore no mobile_device row) exists yet.
event_typeVARCHAR(64)
LICENSE_CHECK / LICENSE_EXCEEDED / CONCURRENT_LIMIT_HIT / LICENSE_RESTORED.
license_limit_hitBOOLEAN NOT NULL
TRUE when this event represents an actual limit breach. Primary filter for "find all license rejections".
concurrent_count nullableINT
How many concurrent sessions were active when this event fired. Shows how close you were to the limit.
max_allowed nullableINT
License cap value at event time. Stored here because limits can change — preserves the historical value for accurate retrospective analysis.
triggered_atTIMESTAMP NOT NULL
When the license event fired. Independent of any session timing.
raw_payload nullableJSONB
Complete original license event from tcomm.
📌 Recommended Indexes
IDXidx_le_limit_hit
(license_limit_hit, triggered_at DESC)
Primary analytics index. Enables "find all license breach events" with time range filtering.
IDXidx_le_null_session (filtered)
triggered_at DESC WHERE session_id IS NULL
Dedicated fast path for pre-session events — the events invisible everywhere else in the schema.
IDX(optional) session_id
Add if needed
A standard session_id index is omitted by default — most license queries filter on license_limit_hit, not session_id. Add if you frequently query "all license events for session X".
Why nullable session_id? When the license cap is hit, tcomm may reject the connection before a CHAT_SESSION row is ever created. If session_id were NOT NULL, these pre-session events could never be recorded. The nullable FK + partial index on WHERE session_id IS NULL creates a dedicated query path for exactly these "invisible" events.
🔌 LEG_DISCONNECT ⭐ NEW · 1:1 per leg · per-participant disconnect ▼

The leg-level equivalent of DISCONNECT_EVENT. Written when any participant's segment ends. Captures when the leg started, when it ended, computed duration, end reason, and license flags — all on one row for instant analytics. Also links to the TRANSFER_EVENT and next leg if the leg ended via handoff.

PK leg_disconnect_idBIGINT
Auto-generated identity.
FK leg_idBIGINT NOT NULL UNIQUE
Links to the SESSION_LEG that ended. UNIQUE — exactly one disconnect record per leg.
FK session_idBIGINT NOT NULL
Denormalized from session_leg for direct "get all leg disconnects for session X" queries without joining through legs.
participant_typeENUM NOT NULL
customer / agent / supervisor / bot. Copied from session_leg — this row is self-contained without a JOIN.
customer_id / agent_id / supervisor_id / bot_id nullableBIGINT
Mirrors the session_leg ID columns. Only one non-NULL, matching participant_type.
leg_startTIMESTAMP NOT NULL
Copied from session_leg.leg_start. Having both start and end on one row makes duration trivial — no JOIN back to session_leg.
leg_endTIMESTAMP NOT NULL
When this participant's segment ended. Set to NOW() when the disconnect fires.
duration_secondsINT COMPUTED
Automatically calculated: (leg_end − leg_start) in seconds. PostgreSQL: EXTRACT(EPOCH…) STORED. SQL Server: DATEDIFF(SECOND,…) PERSISTED. Zero query cost, always accurate.
end_reasonENUM NOT NULL
transferred (handed off) · completed (session ended cleanly) · dropped (connection lost) · license_exceeded · timeout · error.
reason_detail nullableTEXT
Human-readable or machine detail about why this leg ended.
is_license_limitBOOLEAN NOT NULL
TRUE when this specific leg was terminated by a license cap. Scoped to this participant's segment.
tcomm_triggeredBOOLEAN NOT NULL
TRUE when tcomm initiated this leg's termination.
FK transfer_id nullableBIGINT
If end_reason=transferred: points to the TRANSFER_EVENT that caused this leg to end.
FK next_leg_id nullableBIGINT
If end_reason=transferred: points to the SESSION_LEG that replaced this one. Forward navigation through the chain.
raw_payload nullableJSONB
Complete original disconnect signal for this leg.

Why not just use SESSION_LEG.leg_end? SESSION_LEG stores when a leg ended, but not why or what happened. LEG_DISCONNECT is the dedicated disconnect record — the same relationship DISCONNECT_EVENT has with CHAT_SESSION, but at leg granularity. Together: session-level tells you how the whole conversation ended; leg-level tells you each participant's individual story — when they joined, how long they stayed, why they left, and who replaced them.

Use Case Scenarios

How the new SESSION_LEG and TRANSFER_EVENT tables handle real routing situations.

✅ Customer → Bot → Agent COMMON

Customer connects, bot handles first, then transfers to human agent.

  • →CHAT_SESSION created, status=active
  • →LEG #1: type=customer, customer_id=C1, prev=NULL
  • →LEG #2: type=bot, bot_id=B1, prev=Leg1
  • →Messages (Leg2): customer + bot messages, leg_id=Leg2
  • →TRANSFER_EVENT: bot_to_agent, from=Leg2, to=NULL (pending)
  • →LEG #3: type=agent, agent_id=A1, prev=Leg2
  • →TRANSFER_EVENT.to_leg_id updated to Leg3, status=completed
  • →Messages (Leg3): agent messages, leg_id=Leg3
⬆️ Agent Escalates to Supervisor ESCALATION

Agent can't resolve, escalates to supervisor who takes over.

  • →LEG #1: type=customer, customer_id=C1, prev=NULL
  • →LEG #2: type=agent, agent_id=A1, prev=Leg1, status=active
  • →Agent initiates transfer → TRANSFER_EVENT: agent_to_supervisor
  • →LEG #3: type=supervisor, supervisor_id=S1, prev=Leg2
  • →Leg2 status=transferred, Leg3 status=active
  • →Messages from supervisor have leg_id=Leg3
  • →Session ends: DISCONNECT_EVENT, Leg3 status=completed
🔁 Agent → Agent Transfer HANDOFF

First agent transfers to specialist agent. Chain is preserved.

  • →LEG #1: type=customer, customer_id=C1
  • →LEG #2: type=agent, agent_id=A1 (general support)
  • →TRANSFER_EVENT: agent_to_agent, initiated_by=agent
  • →LEG #3: type=agent, agent_id=A2 (specialist), prev=Leg2
  • →Full chain queryable: Leg3 → prev → Leg2 → prev → Leg1
  • →Messages before transfer: leg_id=Leg2. After: leg_id=Leg3
🔴 tcomm License Kill EDGE CASE

License limit fires, tcomm kills session mid-leg.

  • →LICENSE_EVENT inserted (possibly session_id=NULL)
  • →Active leg status=dropped
  • →DISCONNECT_EVENT: is_license_limit=TRUE, tcomm_triggered=TRUE
  • →CHAT_SESSION status=license_exceeded
  • →All messages already saved with leg_id — no data lost
  • →Transfer chain intact even though session ended abruptly
🔍 Query: Full Session Reconstruction ANALYTICS

Reconstruct who handled what, when, and what was said.

  • →JOIN CHAT_SESSION + SESSION_LEG on session_id ORDER BY leg_sequence
  • →For each leg: see participant_type, their ID, start/end times
  • →JOIN TRANSFER_EVENT to see why/how each transfer happened
  • →JOIN CHAT_MESSAGE with leg_id to get messages per participant
  • →JOIN UCID_MAPPING to tie back to telephony record
  • →Full session timeline reconstructable from DB alone
🤖 Failed Transfer FAILURE

Transfer initiated but receiving agent never picks up.

  • →TRANSFER_EVENT inserted: status=pending, to_leg_id=NULL
  • →Timeout fires — no agent accepted
  • →TRANSFER_EVENT.status=failed, completed_at=timeout time
  • →Original leg may resume or session drops
  • →DISCONNECT_EVENT records final outcome
  • →Failed transfers are queryable: WHERE status='failed'

🔍 Common Queries — PostgreSQL
Full session with all legs and transfer chain
SELECT
  s.session_id, s.status, s.start_time,
  sl.leg_id, sl.leg_sequence, sl.participant_type,
  sl.agent_id, sl.bot_id, sl.supervisor_id,
  sl.leg_status, sl.prev_leg_id
FROM chat_session s
JOIN session_leg sl USING (session_id)
WHERE s.session_id = 12345
ORDER BY sl.leg_sequence ASC;
Messages grouped by leg (who said what, when)
SELECT
  sl.leg_sequence, sl.participant_type,
  cm.sequence_no, cm.sender_type, cm.content, cm.sent_at
FROM chat_message cm
JOIN session_leg sl USING (leg_id)
WHERE cm.session_id = 12345
ORDER BY cm.sequence_no ASC;
All transfers for a session with timing
SELECT
  te.transfer_type, te.initiated_by, te.status,
  te.transferred_at, te.completed_at,
  fl.participant_type AS from_type,
  tl.participant_type AS to_type
FROM transfer_event te
JOIN session_leg fl ON te.from_leg_id = fl.leg_id
LEFT JOIN session_leg tl ON te.to_leg_id = tl.leg_id
WHERE te.session_id = 12345;
All sessions handled by a specific agent
SELECT DISTINCT s.session_id, s.start_time, s.status
FROM session_leg sl
JOIN chat_session s USING (session_id)
WHERE sl.agent_id = 99
ORDER BY s.start_time DESC;
Per-participant timeline — who handled what and for how long
SELECT
  sl.leg_sequence, sl.participant_type,
  sl.agent_id, sl.bot_id, sl.supervisor_id,
  ld.leg_start, ld.leg_end,
  ld.duration_seconds,
  ld.end_reason, ld.is_license_limit
FROM session_leg sl
JOIN leg_disconnect ld USING (leg_id)
WHERE sl.session_id = 12345
ORDER BY sl.leg_sequence ASC;
Agent performance — average handle time per agent
SELECT
  agent_id,
  COUNT(*) AS total_legs,
  AVG(duration_seconds) AS avg_duration_sec,
  SUM(CASE WHEN end_reason = 'transferred' THEN 1 ELSE 0 END) AS transferred_count
FROM leg_disconnect
WHERE agent_id IS NOT NULL
  AND leg_end >= NOW() - INTERVAL '7 days'
GROUP BY agent_id
ORDER BY avg_duration_sec DESC;
All tcomm license kills
SELECT s.session_id, s.start_time, d.disconnected_at, d.reason_detail
FROM disconnect_event d
JOIN chat_session s USING (session_id)
WHERE d.is_license_limit = TRUE AND d.tcomm_triggered = TRUE
ORDER BY d.disconnected_at DESC;

    completed_at      DATETIME2     NULL,
    status            NVARCHAR(32)  NOT NULL DEFAULT 'pending'
                                    CONSTRAINT chk_te_status CHECK (
                                      status IN ('pending','completed','failed')),
    CONSTRAINT PK_transfer_event PRIMARY KEY CLUSTERED (transfer_id),
    CONSTRAINT FK_te_session     FOREIGN KEY (session_id)  REFERENCES chat_session(session_id) ON DELETE CASCADE,
    CONSTRAINT FK_te_from_leg    FOREIGN KEY (from_leg_id) REFERENCES session_leg(leg_id),
    CONSTRAINT FK_te_to_leg      FOREIGN KEY (to_leg_id)   REFERENCES session_leg(leg_id)
);
GO
CREATE NONCLUSTERED INDEX IDX_te_session  ON transfer_event (session_id);
CREATE NONCLUSTERED INDEX IDX_te_from_leg ON transfer_event (from_leg_id);
CREATE NONCLUSTERED INDEX IDX_te_to_leg   ON transfer_event (to_leg_id) WHERE to_leg_id IS NOT NULL;

🔍 Key Query Differences — T-SQL
T-SQL: use ON instead of USING, BIT = 1 for true
SELECT s.session_id, sl.leg_sequence, sl.participant_type, sl.agent_id
FROM chat_session s
JOIN session_leg sl ON s.session_id = sl.session_id
WHERE s.session_id = 12345
ORDER BY sl.leg_sequence ASC;

-- License kills: use 1 not TRUE for BIT columns
SELECT s.session_id, d.disconnected_at
FROM disconnect_event d
JOIN chat_session s ON s.session_id = d.session_id
WHERE d.is_license_limit = 1 AND d.tcomm_triggered = 1;
