Session-Centric Database Design

This design places CHAT_SESSION at the center of everything. Every other table — device info, UCID mapping, disconnect events, messages, and license audit — points back to a single session_id. This mirrors how your system works in reality: the session is the first thing created, and all other data is associated to it over time.

🎯 Why CHAT_SESSION Is the Root

session_id Is Generated First

The moment a user initiates a chat, a session_id is created immediately — before the device info is fully captured, before the UCID is known, before any message is sent. Everything else depends on this ID being available, so it must be the root anchor.

All Data Traces Back to It

Every child table uses session_id as its Foreign Key. If you have a message and want to know which device sent it, you go through session_id. If you have a disconnect event and want the UCID, you go through session_id. It is the single lookup key for everything.

Supports Async Data Arrival

UCID is not always available when a session starts — it arrives later from the telephony or CRM system. By making session_id the root, we can create the session record immediately and let UCID_MAPPING link in later without blocking anything.

Survives Abrupt Endings

When tcomm cuts the session due to license limits, there may be no graceful shutdown. Because messages are written to CHAT_MESSAGE with session_id continuously, they are already saved. The session record exists. No data is lost even if cleanup code never runs.

Simple Querying

Want everything about a session? One WHERE clause: WHERE session_id = '...' across all tables. You do not need complex joins through intermediate tables. The star schema around session_id keeps queries simple and fast.

Independent Audit Trail

LICENSE_EVENT can record license violations even when no session_id exists yet (nullable FK). This means even the edge case where a license limit fires before the session is fully created is captured independently, closing every audit gap.

📐 Key Design Decisions
DECISION WHAT WE DID WHY
ROOT TABLE
CHAT_SESSION as anchor		 session_id is a UUID generated at the very start of a chat session_id is the first known piece of data. All other data arrives later and references it. Making it the root avoids any chicken-and-egg dependency problem.
ASYNC PATTERN
UCID resolved later		 UCID_MAPPING row inserted as a separate step when CRM/telephony provides the UCID value UCID is not always available at session start. Separating it into its own table means session creation is never blocked waiting for UCID. The session_id FK links them whenever UCID arrives.
NULLABLE FK
LICENSE_EVENT.session_id is nullable		 session_id column in LICENSE_EVENT allows NULL values A license limit can fire before a session record is created (e.g., the connection is refused instantly). Making session_id nullable lets us record this event even with no session to point to.
SEQUENCE ORDER
sequence_no in CHAT_MESSAGE		 Each message gets an incrementing integer sequence_no within a session Timestamps can be identical or arrive slightly out of order in high-throughput systems. sequence_no guarantees the exact order of conversation can always be reconstructed.
RAW STORAGE
JSONB raw_payload on 3 tables		 raw_payload JSONB column on CHAT_MESSAGE, DISCONNECT_EVENT, and LICENSE_EVENT Saves the complete original payload from the source system. If the application parsing logic has a bug, the raw data is still there. Enables re-processing without data loss.
BOOLEAN FLAGS
is_license_limit, tcomm_triggered		 Explicit boolean columns on DISCONNECT_EVENT instead of relying on reason_code parsing Filtering "all sessions that ended due to license" with a WHERE clause on a boolean is fast and unambiguous. Parsing reason_code strings is error-prone and slow at scale.
STATUS ENUM
status on CHAT_SESSION		 ENUM with values: active, ended, dropped, license_exceeded A quick glance at CHAT_SESSION.status tells you how the session ended without needing to JOIN to DISCONNECT_EVENT. Useful for dashboards and summary queries.

Table-by-Table Deep Dive

Each table below is explained field-by-field with the reasoning behind each column's type, constraints, and purpose. Click any table to expand it.

💬 CHAT_SESSION ROOT TABLE ▼

This is the anchor of the entire schema. A row is created here the instant a chat begins. It has no foreign keys pointing outward — every other table points inward to this one. Its session_id is the single key used to retrieve all associated data: device, UCID, messages, disconnect reason, and license events.

PK session_id UUID
Auto-generated unique identifier. UUID is used instead of an integer to avoid sequential guessing and to support distributed generation without a central counter.
start_time TIMESTAMP
Exact moment the chat session was initiated. Used to calculate session duration and for time-based reporting.
end_time TIMESTAMP / NULL
NULL while session is active. Set when the session ends for any reason. If tcomm kills the session abruptly, this may be set by a cleanup job after the fact.
status ENUM
Current state: active (ongoing), ended (clean close), dropped (connection lost), license_exceeded (tcomm cut it). Allows fast dashboard queries without joining DISCONNECT_EVENT.
channel VARCHAR(32)
The communication channel: e.g. 'mobile_app', 'web', 'api'. Useful for segmenting analytics by entry point.
metadata JSONB
Flexible key-value store for any additional session-level context (e.g. region, agent_id, queue_name). Avoids schema changes when new attributes are needed.
Design note: CHAT_SESSION intentionally has no foreign keys. It is fully self-contained. All dependencies flow outward from it, not into it. This is what makes it the root — deleting a session cascades down, not up.
📱 MOBILE_DEVICE IDENTITY · 1:1 ▼

Captures the identity of the physical device at the moment a session starts. Written simultaneously with CHAT_SESSION creation. The mobile_id (the ID your mobile app generates or assigns) is stored here with a UNIQUE constraint so it can be used as a lookup key even without knowing the session_id.

PK device_id UUID
Internal primary key for this device record. Separate from mobile_id so the internal DB key is always system-controlled.
FK session_id UUID
Links this device record to its session. This FK is what makes MOBILE_DEVICE a child of CHAT_SESSION.
UK mobile_id VARCHAR(64)
The mobile identifier from your app (could be IMEI, push token, or app-generated UUID). UNIQUE constraint ensures no duplicates. Can be used to find all sessions from a specific device.
platform VARCHAR(32)
iOS, Android, etc. Useful for filtering device-specific issues.
device_fingerprint VARCHAR(128)
A hash combining device hardware identifiers. Useful for detecting the same physical device across different mobile_id values (e.g. after app reinstall).
captured_at TIMESTAMP
Exact time device info was recorded. Should match or be very close to CHAT_SESSION.start_time.
Design note: In the previous v1 design, MOBILE_DEVICE was the root and sessions hung off devices. Now device info hangs off sessions — meaning the same physical device (same mobile_id) will have multiple rows in this table across different sessions. To find all sessions from one device, query: WHERE mobile_id = '...'
🔗 UCID_MAPPING ASYNC LINK · 1:1 ▼

The asynchronous bridge between your session and the telephony/CRM system's UCID. This row does not exist at session start — it is inserted later when the external system provides the UCID. The session_id FK is how it links back. This table also cross-references mobile_id directly for convenience, so you can find the UCID from either the session or the device.

PK ucid_map_id UUID
Internal primary key for this mapping record.
FK session_id UUID
Links this UCID back to the chat session it belongs to. This is how you go from UCID → session.
UK ucid_value VARCHAR(128)
The actual UCID string from the telephony/CRM system. UNIQUE constraint ensures one UCID maps to exactly one session. Prevents accidental duplicate mappings.
mobile_id VARCHAR(64)
Denormalized copy of mobile_id for direct lookup convenience. Lets you find "which device had UCID X" without an extra JOIN through CHAT_SESSION → MOBILE_DEVICE.
resolved_at TIMESTAMP
When the UCID was actually resolved. The gap between session start_time and resolved_at shows how long UCID resolution took.
resolution_status ENUM
Values like 'resolved', 'failed', 'pending'. Allows querying sessions where UCID was never resolved — important for troubleshooting lost records.
Design note: Because UCID arrives asynchronously, there will be sessions where this table has no corresponding row. That is normal and expected. A LEFT JOIN is used when querying: SELECT * FROM CHAT_SESSION LEFT JOIN UCID_MAPPING USING (session_id). Sessions with no UCID row are simply not yet resolved.
⚡ DISCONNECT_EVENT REASON RECORD · 1:1 ▼

Records the exact reason a session ended. Written once per session at termination time. The two boolean flags — is_license_limit and tcomm_triggered — are the most critical fields for your use case, as they directly identify sessions that were silently killed by tcomm due to license overflow.

FK session_id UUID
Links this disconnect record to its session. One session → one disconnect record (1:1 relationship).
reason_code VARCHAR(64)
A short code describing the disconnect reason. Examples: LICENSE_EXCEEDED, USER_CLOSED, AGENT_ENDED, NETWORK_DROP, TIMEOUT. Used for grouping and reporting.
reason_detail TEXT
Full human-readable description of the disconnect. Could be the error message, stack trace summary, or system log entry. No length limit.
is_license_limit BOOLEAN
Key field. TRUE when the session was terminated specifically because a license limit was reached. Enables instant filtering without parsing reason_code strings.
tcomm_triggered BOOLEAN
Key field. TRUE when tcomm itself initiated the disconnect (as opposed to the user, agent, or app). The combination of both flags TRUE identifies the "silent license kill" scenario.
raw_payload JSONB
The complete original disconnect signal/event from tcomm or the platform. Preserves every detail even if the application parsing logic missed something.
To find all silent license kills: SELECT * FROM DISCONNECT_EVENT WHERE is_license_limit = TRUE AND tcomm_triggered = TRUE. Two boolean columns make this query instant and unambiguous.
✉️ CHAT_MESSAGE RAW DATA · 1:N ▼

The high-volume raw message store. Every single message — from user, agent, or bot — is written here as it arrives, in real-time. This table will be by far the largest in the system. Each row carries both the parsed fields (content, sender_type) and the complete original raw_payload JSONB, ensuring nothing is ever lost.

FK session_id UUID
Links every message to its session. The compound index on (session_id, sequence_no) is the most important index in this table.
sequence_no INT
Auto-incrementing integer scoped per session (1, 2, 3...). Guarantees conversation order is always recoverable regardless of timestamp accuracy.
sender_type ENUM
'user', 'agent', or 'bot'. Quickly separates human messages from automated responses without parsing content.
sender_id VARCHAR(64)
The specific ID of whoever sent the message — mobile_id for users, agent_id for agents, bot_id for bots. Allows filtering all messages by a specific participant.
content TEXT
The human-readable message text. Extracted from raw_payload for easy searching and display. NULL for non-text message types (files, images, events).
raw_payload JSONB
The complete original message object. Preserves every field the source system sent — attachments, metadata, delivery receipts, custom fields. Never loses data.
sent_at / received_at TIMESTAMP
Two separate timestamps: when the sender sent it, and when the system received it. The gap between them reveals network latency and delivery delays.
Performance note: This table grows fastest. Consider partitioning by sent_at (monthly partitions) for large deployments. The compound index (session_id, sequence_no) covers the most common query pattern of fetching a full conversation in order.
🔑 LICENSE_EVENT AUDIT · 1:N / STANDALONE ▼

An independent audit log for all license-related events. Unlike all other child tables, session_id is nullable here. This is intentional — a license limit can fire before a session record is created (when the system refuses a new connection instantly). This table captures every license event regardless of whether a session exists to link to.

FK session_id UUID / NULL
Nullable. Points to a session if one exists. NULL when the license limit fires before the session record was created — this is the edge case that makes this column nullable.
mobile_id VARCHAR(64) / NULL
Also nullable. Captured directly here (not via JOIN) so license events can be traced to a device even when session_id is NULL.
event_type VARCHAR(64)
The type of license event: e.g. LICENSE_CHECK, LICENSE_EXCEEDED, LICENSE_RESTORED, CONCURRENT_LIMIT_HIT. Allows filtering by event category.
license_limit_hit BOOLEAN
TRUE when the event resulted in a limit being reached. FALSE for informational checks that passed. Enables quick count of actual limit violations.
concurrent_count INT
The number of concurrent sessions at the time of this event. Together with max_allowed, shows how close to the limit the system was.
max_allowed INT
The license limit at the time of this event. Stored here because license limits can change over time — capturing the value at event time preserves historical accuracy.
Design note: This table solves the core problem you described — "if chat ends at tcomm due to license, there is no record anywhere." LICENSE_EVENT exists specifically to ensure there is always a record. Even if tcomm kills the connection before any other table gets written, this event is captured first, independently, with a nullable session_id.

Real-World Use Case Scenarios

How data flows through the database in different real situations — normal sessions, license kills, async UCID resolution, and incomplete sessions.

✅ Normal Chat Session HAPPY PATH

User opens app, chats with agent, session ends cleanly. All data arrives in sequence.

  • → CHAT_SESSION created (status=active, end_time=NULL)
  • → MOBILE_DEVICE row inserted with session_id FK
  • → CHAT_MESSAGE rows written as messages arrive (sequence 1,2,3...)
  • → UCID_MAPPING row inserted when CRM resolves UCID
  • → Session ends → DISCONNECT_EVENT written (reason=USER_CLOSED)
  • → CHAT_SESSION updated: status=ended, end_time set
🔴 tcomm License Kill YOUR KEY CASE

System hits license cap, tcomm kills the connection with no warning. Previously this left no record.

  • → LICENSE_EVENT inserted immediately (license_limit_hit=TRUE, session_id may be NULL)
  • → CHAT_SESSION created (status=active) if connection got that far
  • → MOBILE_DEVICE and partial CHAT_MESSAGE rows may exist
  • → tcomm cuts connection → DISCONNECT_EVENT written (is_license_limit=TRUE, tcomm_triggered=TRUE)
  • → Cleanup job updates CHAT_SESSION status=license_exceeded, sets end_time
  • → UCID_MAPPING may never arrive — that's OK, row is simply absent
⏳ Async UCID Resolution DELAYED DATA

Session starts, ends, then hours later the CRM provides the UCID for that session.

  • → CHAT_SESSION created (no UCID yet)
  • → All messages and device info stored normally
  • → Session ends → DISCONNECT_EVENT written
  • → Hours/days later: CRM webhook fires with UCID value + session_id
  • → UCID_MAPPING row inserted (resolved_at = now, not session start time)
  • → Full traceability now available: UCID ↔ session ↔ device ↔ messages
📵 Network Drop MID-SESSION

User loses connectivity mid-conversation. Session ends abruptly without user action.

  • → CHAT_SESSION active, messages being written normally
  • → Network drops — no graceful close signal received
  • → Heartbeat timeout detected by server
  • → DISCONNECT_EVENT written (reason_code=NETWORK_DROP, tcomm_triggered=FALSE)
  • → CHAT_SESSION updated: status=dropped, end_time=timeout timestamp
  • → All messages up to last received are safely stored in CHAT_MESSAGE
🚫 Pre-Session License Block EDGE CASE

License limit is hit before the session record is even created. Previously invisible.

  • → Mobile app sends connection request with mobile_id
  • → System checks license — limit already reached
  • → LICENSE_EVENT inserted instantly (session_id=NULL, mobile_id captured)
  • → Connection refused — no CHAT_SESSION row is ever created
  • → No MOBILE_DEVICE, CHAT_MESSAGE, or DISCONNECT_EVENT rows
  • → The LICENSE_EVENT is the only proof this attempt happened ← solves your problem
🔍 Audit Query: Find Lost Sessions REPORTING

Operations team wants to find all sessions killed by license limits that have no UCID resolved.

  • → JOIN CHAT_SESSION + DISCONNECT_EVENT on session_id
  • → Filter: is_license_limit=TRUE AND tcomm_triggered=TRUE
  • → LEFT JOIN UCID_MAPPING — sessions with no UCID row are unresolved
  • → Also check LICENSE_EVENT where session_id IS NULL for pre-session blocks
  • → Result: complete list of affected sessions with device info and message history
  • → Compare concurrent_count vs max_allowed in LICENSE_EVENT for capacity analysis

CREATE TABLE license_event (

CREATE INDEX idx_le_null_session ON license_event (triggered_at DESC) WHERE session_id IS NULL;
🔍 Common Queries — PostgreSQL
Full session — all tables joined (PostgreSQL USING syntax)
SELECT
  s.session_id, s.status, s.start_time, s.end_time,
  d.mobile_id, d.platform, d.app_version,
  u.ucid_value, u.resolved_at,
  de.reason_code, de.is_license_limit, de.tcomm_triggered
FROM chat_session s
LEFT JOIN mobile_device    d  USING (session_id)
LEFT JOIN ucid_mapping     u  USING (session_id)
LEFT JOIN disconnect_event de USING (session_id)
WHERE s.session_id = 'your-uuid';
All messages for a session in order
SELECT sequence_no, sender_type, sender_id, content, sent_at
FROM chat_message
WHERE session_id = 'your-uuid'
ORDER BY sequence_no ASC;
All tcomm license kills
SELECT s.session_id, s.start_time, d.mobile_id, de.disconnected_at
FROM chat_session s
JOIN  disconnect_event de USING (session_id)
LEFT JOIN mobile_device  d  USING (session_id)
WHERE de.is_license_limit = TRUE AND de.tcomm_triggered = TRUE
ORDER BY de.disconnected_at DESC;
Pre-session license blocks

🏗️ Create Tables — MS SQL Server 2016+ (T-SQL)
MS SQL Server specifics:  UNIQUEIDENTIFIER · NVARCHAR(MAX) for JSON · BIT (0/1) · SYSDATETIME() · NEWSEQUENTIALID() · ON in JOINs (no USING) · GO batch separator
💬 CHAT_SESSION ⚓ ROOT No FK outward. Generated first. Everything references this. 
CREATE TABLE chat_session (
    session_id   UNIQUEIDENTIFIER NOT NULL DEFAULT NEWSEQUENTIALID(),
    start_time   DATETIME2        NOT NULL DEFAULT SYSDATETIME(),
    end_time     DATETIME2        NULL,
    status       NVARCHAR(32)     NOT NULL DEFAULT 'active'
                                  CONSTRAINT chk_cs_status CHECK (
                                    status IN ('active','ended','dropped','license_exceeded')),
    channel      NVARCHAR(32)     NULL,
    created_at   DATETIME2        NOT NULL DEFAULT SYSDATETIME(),
    metadata     NVARCHAR(MAX)    NULL,    -- JSON stored as NVARCHAR(MAX)
    CONSTRAINT PK_chat_session PRIMARY KEY CLUSTERED (session_id)
);
GO
CREATE NONCLUSTERED INDEX IDX_cs_status    ON chat_session (status);
CREATE NONCLUSTERED INDEX IDX_cs_created   ON chat_session (created_at DESC);
CREATE NONCLUSTERED INDEX IDX_cs_status_ts ON chat_session (status, created_at DESC);
GO

CREATE TABLE license_event (

🔍 Common Queries — MS SQL Server
Full session — T-SQL uses ON instead of USING
SELECT
  s.session_id, s.status, s.start_time, s.end_time,
  d.mobile_id, d.platform, d.app_version,
  u.ucid_value, u.resolved_at,
  de.reason_code, de.is_license_limit, de.tcomm_triggered
FROM chat_session s
LEFT JOIN mobile_device    d  ON s.session_id = d.session_id
LEFT JOIN ucid_mapping     u  ON s.session_id = u.session_id
LEFT JOIN disconnect_event de ON s.session_id = de.session_id
WHERE s.session_id = 'your-guid';
All messages for a session in order
SELECT sequence_no, sender_type, sender_id, content, sent_at
FROM chat_message
WHERE session_id = 'your-guid'
ORDER BY sequence_no ASC;
All tcomm license kills — BIT uses 1 not TRUE
SELECT s.session_id, s.start_time, d.mobile_id, de.disconnected_at
FROM chat_session s
JOIN  disconnect_event de ON s.session_id = de.session_id
LEFT JOIN mobile_device  d  ON s.session_id = d.session_id
WHERE de.is_license_limit = 1 AND de.tcomm_triggered = 1
ORDER BY de.disconnected_at DESC;
